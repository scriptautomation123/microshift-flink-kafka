# Project Development Standards

This guide documents all standards for documentation, code, configuration, deployment, and testing across the MicroShift + Kafka + Flink platform.

---

## 1. Documentation Standards

**→ `README.md` — The ONLY authoritative documentation source**

### Single Source of Truth

This is the official documentation standard for this project going forward:
- All procedures documented in README.md only
- No duplication across multiple files
- All updates made to README.md
- All navigation, TOC, and cross-links in README.md
- Backup of consolidated files preserved in `docs/.backup/`

### When Updating Documentation

1. **Edit only README.md** — Find section via Table of Contents
2. **Make changes directly** — In that section, no file splitting
3. **Update Table of Contents** — If adding new major sections
4. **Test commands** — Before committing
5. **Use markdown anchors** — For internal links: `[text](#heading-lowercased)`

## 2. Bash Script Standards

### Location & Organization

**Shared utilities** (root level):
- **Location**: Root directory (`.`)
- **Contents**: `lib.sh` (common functions), `microshift.sh` (MicroShift operations)
- **Usage**: Sourced by component scripts via `source "$(cd "$(dirname "$0")/../.." && pwd)/lib.sh"`

**Component-specific scripts**:
- **Kafka**: Component functions integrated into root `lib.sh` (no separate scripts)
- **Flink**: Component functions integrated into root `lib.sh` (no separate scripts)
- **MicroShift**: Consolidated into single script (`microshift.sh` at root level)
- **Runbooks/Orchestration**: `runbooks/scripts/` (test-all, cleanup-all, validate-platform-all.sh)

### Naming Convention

- **Format**: `verb-noun.sh` (e.g., `build-image.sh`, `deploy.sh`, `check.sh`)
- **Executability**: Always `chmod +x *.sh`
- **No shebang in lib.sh**: `lib.sh` is sourced, not executed directly

### Structure Template

**For component scripts** (if they existed in kafka/openshift/scripts, flink/openshift/scripts):

```bash
#!/usr/bin/env bash
set -euo pipefail

# Set BUNDLE_DIR and RENDER_DIR for lib.sh (component-specific context)
export BUNDLE_DIR="$(cd "$(dirname "$0")/../.." && pwd)"
export RENDER_DIR="${BUNDLE_DIR}/.rendered"

# Source shared utilities from root lib.sh
source "$(cd "$(dirname "$0")/../../.." && pwd)/lib.sh"

usage() {
  cat <<'EOF'
Usage: script-name [OPTIONS]
  --option1  Description
  --option2  Description
Examples:
  script-name --option1 value
EOF
}

while [[ $# -gt 0 ]]; do
  case "$1" in
    -h|--help) usage; exit 0 ;;
    --option1) VAR1="$2"; shift 2 ;;
    *) echo "unknown: $1" >&2; usage; exit 1 ;;
  esac
done

main() {
  # implementation using shared lib functions
  load_env "${ENV_FILE}"
  require_env REQUIRED_VARS
  true
}

main "$@"
```

**For runbook/platform scripts** (root-level orchestration):

```bash
#!/usr/bin/env bash
set -euo pipefail

# Source shared utilities from root lib.sh
source "$(cd "$(dirname "$0")/.." && pwd)/lib.sh"

usage() {
  cat <<'EOF'
Usage: script-name [OPTIONS]
EOF
}

main() {
  # platform-level logic
  true
}

main "$@"
```

### Requirements

- ✓ Shebang: `#!/usr/bin/env bash` (component scripts only; lib.sh omits shebang)
- ✓ Error handling: `set -euo pipefail`
- ✓ Set BUNDLE_DIR/RENDER_DIR: Before sourcing lib.sh (component scripts)
- ✓ Source shared lib: `source "$(cd "$(dirname "$0")/../../.." && pwd)/lib.sh"`
- ✓ Usage function with examples
- ✓ Quote all variables: `"${VAR}"` not `$VAR`
- ✓ Errors to stderr: `echo "error" >&2`
- ✓ Exit 0 (success) or 1 (error)
- ✓ No hardcoded paths (use relative or parameterized)
- ✓ Fail fast; don't recover with `||`
- ✓ Use shared lib functions: `load_env()`, `require_env()`, `log()`, `die()`, `detect_container_cli()`

### Shared Library Functions (lib.sh)

**Logging & Errors**:
- `log "message"` — Log with timestamp to stdout
- `die "message"` — Log error and exit with code 1

**Requirements & Validation**:
- `require_commands cmd1 cmd2` — Exit if commands not found
- `require_env VAR1 VAR2` — Exit if environment variables empty
- `require_file /path/to/file` — Exit if file doesn't exist

**Environment & Container**:
- `load_env <env-file>` — Source env file with error checking
- `detect_container_cli` — Return podman or docker path

**Template Rendering** (Flink-specific):
- `render_template <src> <dest>` — Render {{ VAR }} templates
- `render_tree <path>...` — Render directory trees
- `render_bundle` — Full bundle rendering (Flink conf/manifests/sql)
- `ensure_render_dir` — Create .rendered/ directory

**Generic Deployment Helpers** (Kubernetes operations):
- `create_namespace <namespace>` — Create namespace if missing, switch to it
- `apply_manifests <namespace> <file1> [<file2>...]` — Apply manifests with logging
- `create_docker_registry_secret <namespace> <image> <secret-name> <service-account>` — Create/update docker-registry secret and link to SA
- `wait_for_deployment <namespace> <name>` — Wait for Deployment readiness (10m timeout)
- `wait_for_statefulset <namespace> <name>` — Wait for StatefulSet readiness (10m timeout)
- `apply_from_template <template-file> <namespace>` — Process OpenShift template and apply resources

**Utilities**:
- `json_get <file> <key>` — Extract JSON values
- `sql_gateway_api_url` — Get Flink SQL Gateway endpoint
- `rendered_file <relative-path>` — Get rendered file path
- `secret_name_or_default <name> <fallback>` — Safe secret naming

### Common Patterns

**Shared utilities** (root level):

| Script | Location | Purpose | Used By |
|--------|----------|---------|---------|
| `lib.sh` | Root directory (`.`) | Utility functions (log, die, require_*, load_env, detect_container_cli, render_template, create_namespace, apply_manifests, create_docker_registry_secret, wait_for_*, apply_from_template) | All component scripts |
| `microshift.sh` | Root directory (`.`) | MicroShift cluster management | Manual or runbook execution |

**Generic Deployment Helpers** (in lib.sh):
These consolidate common Kubernetes operations:

| Helper | Purpose | Example |
|--------|---------|---------|
| `create_namespace` | Create/switch to namespace | `create_namespace "${OPENSHIFT_NAMESPACE}"` |
| `apply_manifests` | Apply multiple manifests with logging | `apply_manifests "${NS}" "$(rendered_file 00-sa.yaml)" "$(rendered_file 01-cm.yaml)"` |
| `create_docker_registry_secret` | Create & link pull secret | `create_docker_registry_secret "${NS}" "${IMAGE}" "ghcr-pull" "runner-sa"` |
| `wait_for_deployment` | Wait for Deployment readiness | `wait_for_deployment "${NS}" my-deployment` |
| `wait_for_statefulset` | Wait for StatefulSet readiness | `wait_for_statefulset "${NS}" my-statefulset` |
| `apply_from_template` | Process & apply OpenShift template | `apply_from_template "template.yaml" "${NS}"` |

**Component-specific scripts**:

| Script | Location | Purpose | Usage |
|--------|----------|---------|-------|
| `render-manifests.sh` | Kafka scripts | Render manifest templates to `.rendered/manifests/` | `render-manifests.sh env/dev.env` |
| `render.sh` | Flink scripts | Render manifest templates to `.rendered/manifests/` | `render.sh env/dev.env` |
| `apply-manifests.sh` | Kafka/Flink scripts | Apply OpenShift template (Flink) or rendered manifests (Kafka) | `apply-manifests.sh env/dev.env` |
| `deploy.sh` | Kafka/Flink scripts | Orchestrate: render → apply | `deploy.sh env/dev.env` |
| `build-image.sh` / `build-images.sh` | Kafka/Flink scripts | Build container images | `build-image.sh env/dev.env` |
| `delete.sh` | Kafka scripts | Delete all resources | `delete.sh env/dev.env` |
| `check.sh` | Kafka scripts | Health checks | `check.sh env/dev.env` |
| `create-topic.sh` | Kafka scripts | Create Kafka topics | `create-topic.sh env/dev.env --topic name` |
| `*-all-functionality.sh` | All components | End-to-end tests | `test-all-functionality.sh` |
| `validate-*.sh` | Kafka/Flink scripts | Validation checks | `validate-namespace-identities.sh` |
| `submit-sql.sh` | Flink scripts | Submit SQL jobs | `submit-sql.sh env/dev.env` |

**Platform-level scripts** (runbooks/scripts/):

| Script | Location | Purpose | Usage |
|--------|----------|---------|-------|
| `test-all` | `runbooks/scripts/` | Full platform deployment + validation | `test-all --clean --kafka-env ... --flink-env ...` |
| `validate-platform-all.sh` | `runbooks/scripts/` | Non-destructive health check | `validate-platform-all.sh [--json]` |

## 3. Dockerfile Standards

### Location & Naming

- **Kafka**: `kafka-docker/Dockerfile.kraft`
- **Flink**: `flink-docker/Dockerfile.base`, `Dockerfile.sql-runtime`

### Requirements

- ✓ Base: UBI9 only (`registry.access.redhat.com/ubi9/ubi*`)
- ✓ Non-root user: `USER appuser` (UID 1000+)
- ✓ Multi-stage: Build + runtime stages
- ✓ Minimal: Use `ubi-minimal` for runtime
- ✓ Documented: Comments on non-obvious config
- ✓ Startup script: Explicit path, not direct command

### Template

```dockerfile
FROM registry.access.redhat.com/ubi9/ubi as builder
RUN yum install -y openssl
COPY . /build/
RUN cd /build && make

FROM registry.access.redhat.com/ubi9/ubi-minimal
RUN microdnf install -y openssl
RUN useradd -u 1000 -r appuser && chown -R appuser /opt/app
COPY --from=builder /build/dist /opt/app
USER appuser
ENTRYPOINT ["/opt/app/startup.sh"]
```

## 4. YAML Manifest Standards

### Location & Numbering

- **Kafka**: `manifests/` (search for kafka-related files like 00-serviceaccount.yaml, 03-statefulset.yaml)
- **Flink**: `manifests/` (search for flink-related files like 00-serviceaccount-rbac.yaml, 03-jobmanager.yaml, 04-taskmanager.yaml)

### Deployment Order (Strict)

1. **00** — ServiceAccounts, RBAC
2. **01** — ConfigMaps
3. **02** — Secrets
4. **03** — Services
5. **04** — StatefulSets, PodDisruptionBudgets
6. **06** — Routes/Ingress
7. **07-10** — RBAC governance

### Requirements

- ✓ Templates: Use `{{ VAR }}` for substitution
- ✓ One resource per file (except closely related)
- ✓ Labels: `app.kubernetes.io/name`, `app.kubernetes.io/version`
- ✓ Images: Use `{{ IMAGE_VAR }}` (never hardcode)
- ✓ Namespaces: Use `{{ OPENSHIFT_NAMESPACE }}` variable
- ✓ Non-root: `runAsNonRoot: true`, `allowPrivilegeEscalation: false`
- ✓ Resources: requests/limits defined for all containers
- ✓ ServiceAccount: Explicitly specified (never default)

### Example

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: {{ COMPONENT }}-runner
  namespace: {{ OPENSHIFT_NAMESPACE }}
  labels:
    app.kubernetes.io/name: {{ COMPONENT }}
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: {{ COMPONENT }}
  namespace: {{ OPENSHIFT_NAMESPACE }}
spec:
  serviceName: {{ COMPONENT }}-headless
  replicas: 3
  selector:
    matchLabels:
      app: {{ COMPONENT }}
  template:
    spec:
      serviceAccountName: {{ COMPONENT }}-runner
      securityContext:
        runAsNonRoot: true
        fsGroup: 65534
      containers:
      - name: {{ COMPONENT }}
        image: {{ COMPONENT_IMAGE }}
        resources:
          requests:
            memory: "1Gi"
            cpu: "250m"
```

## 5. SQL File Standards

### Location & Numbering

- **Location**: `flink-sql/` (10-session-config.sql, 20-kafka-source.sql, 30-kafka-sink.sql, 40-pipeline.sql)
- **Pattern**: `10-session-config.sql`, `20-kafka-source.sql`, `30-kafka-sink.sql`, `40-pipeline.sql`

### Execution Order

1. **10** — Session config (parallelism, timeout)
2. **20** — Source table
3. **30** — Sink table
4. **40** — Pipeline (INSERT statement)

### Requirements

- ✓ Comments: Explain table purpose and schema
- ✓ Idempotent: Use `CREATE OR REPLACE`
- ✓ Naming: PascalCase tables (e.g., `KafkaSource`)
- ✓ Properties: Document Kafka host, topic, format
- ✓ Exactly-once: Enable for critical sinks

### Example

```sql
-- Table: KafkaSource
-- Purpose: Read events from ha-drill topic
CREATE OR REPLACE TABLE KafkaSource (
  id STRING,
  ts BIGINT,
  message STRING
) WITH (
  'connector' = 'kafka',
  'topic' = 'ha-drill',
  'properties.bootstrap.servers' = '{{ KAFKA_BROKER_HOST }}:{{ KAFKA_BROKER_PORT }}',
  'format' = 'json'
);
```

## 6. Configuration & Environment Standards

### Location & Naming

- **Kafka**: `env/kafka.dev.env` (previously `kafka/openshift/env/`)
- **Flink**: `env/flink.dev.env` (previously `flink/openshift/env/`)
- **Configuration**: `env/flink.config.yaml` (Flink SQL runtime configuration)
- **.gitignore**: Never commit runtime .env files

### Variable Naming

- **Format**: `UPPERCASE_WITH_UNDERSCORES`
- **Prefix**: `KAFKA_*`, `FLINK_*`, `OPENSHIFT_*`, `REGISTRY_*`
- **Examples**: `KAFKA_IMAGE=ghcr.io/owner/kafka:3.7.1`, `FLINK_NAMESPACE=flink-dev`

### Requirements

- ✓ example.env: All required variables documented
- ✓ Comments: Describe each variable
- ✓ No secrets: Use placeholders (`<your-username>`)
- ✓ Defaults: Meaningful or empty, never hardcoded production values
- ✓ Grouped: Related variables with section comments

## 7. Deployment Workflow Standards

### Standard Sequence (Kafka & Flink)

All Kafka and Flink operations are now provided by root-level `lib.sh` functions. Usage is typically through orchestration scripts in `runbooks/scripts/` or direct invocation:

```bash
# Source lib.sh and call functions directly
source lib.sh

# Kafka operations
kafka_deploy "${NAMESPACE}" "env/kafka.dev.env"
kafka_create_topic my-topic 3

# Flink operations
flink_deploy "${NAMESPACE}" "env/flink.dev.env"
flink_submit_sql "env/flink.dev.env"

# Platform orchestration (automated)
runbooks/scripts/test-all --clean --kafka-env env/kafka.dev.env --flink-env env/flink.dev.env
```

**All components** now use:
- `lib.sh` — Single source for all operations (render, apply, deploy, test, cleanup)
- `env/` — Centralized configuration (kafka.dev.env, flink.dev.env, etc.)
- `manifests/` — All Kubernetes resources (Kafka and Flink mixed)
- `flink-docker/` — Flink container images
- `kafka-docker/` — Kafka container images
- `flink-sql/` — SQL definitions for Flink jobs

## 8. Testing Standards

### Test Script Pattern

```bash
#!/usr/bin/env bash
set -euo pipefail

PASS=0; FAIL=0; WARN=0

run_check() {
  local name="$1"; shift
  set +e; "$@"; local rc=$?; set -e
  if [[ $rc -eq 0 ]]; then
    echo "PASS: ${name}"
    PASS=$((PASS+1))
  else
    echo "FAIL: ${name} (rc=${rc})"
    FAIL=$((FAIL+1))
  fi
}

run_check "description" command arg1 arg2

echo "Summary: PASS=${PASS} FAIL=${FAIL} WARN=${WARN}"
[[ ${FAIL} -eq 0 ]] && exit 0 || exit 1
```

### Test Scripts

| Script | Location | Purpose |
|--------|----------|---------|
| `test-all-functionality.sh` | `*/openshift/scripts/` | Component integration test |
| `test-all` | `runbooks/scripts/` | Full platform deployment |
| `validate-platform-all.sh` | `runbooks/scripts/` | Non-destructive health check |
| `cleanup-all` | `runbooks/scripts/` | Complete teardown |

### Test Flags

- **`--clean`** — Delete resources after test
- **`--strict`** — Fail on warnings, not just errors
- **`--json`** — JSON output for CI/CD parsing
- **`--skip-runtime`** — Skip cluster checks (offline mode)

### Validation Checks

1. Bash syntax: `bash -n script.sh`
2. Executability: `chmod +x *.sh`
3. Manifest validation: `oc apply --dry-run=client`
4. Configuration: All env vars present
5. Security: No hardcoded credentials
6. Runtime: Pod health and readiness
7. Functional: End-to-end data flow

## Key Principles (All Standards)

✓ **Single Source of Truth** — README.md for docs; numbered manifests/scripts for deployment  
✓ **Consolidated Utilities** — Shared lib.sh in root scripts/ folder, sourced by all components  
✓ **No Duplication** — Each procedure documented once; each script does one job; utilities shared  
✓ **Fail Fast** — Clear error messages, non-zero exit codes  
✓ **Non-Root** — All containers run as non-root  
✓ **Idempotent** — Safe to run multiple times  
✓ **Automatable** — JSON output for CI/CD  
✓ **Documented** — Comments in code, examples in README  
✓ **Maintainable** — Clear naming, consistent patterns
